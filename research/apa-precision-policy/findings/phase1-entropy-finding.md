# Entropy Is All You Need: Evolved Mixed-Precision Policy for FlashAttention

**One-line summary:** Evolutionary search over 80 iterations and 14 candidate features converged on a two-threshold policy -- use INT8 everywhere except blocks with outliers AND entropy < 2.0 -- compressing 58% of attention blocks with no accuracy loss.

## Background

FlashAttention computes exact attention in tiled blocks that never materialize the full N x N score matrix, trading recomputation for memory efficiency. Mixed-precision attention extends this by assigning different numerical precisions to different blocks: blocks where quantization error is tolerable use fewer bits (INT8 at 8 bits), while sensitive blocks retain full precision (FP16 at 16 bits). The challenge is deciding which blocks need protection. A wrong decision on outlier-heavy blocks causes catastrophic accuracy loss (1000x worse RMSE), while unnecessary FP16 wastes memory bandwidth.

## Method

We used [OpenEvolve](https://github.com/codelion/openevolve), an evolutionary code search framework, to discover the optimal precision policy function. The search operated as follows:

- **What was evolved:** A pure-Python function `precision_policy(block_stats) -> precision` mapping 14 per-block statistics to one of six precision levels (fp16, fp8_e4m3, fp8_e5m2, int8, fp4, int4).
- **Input features (14):** q_norm, k_norm, v_norm, score_max, score_mean, score_var, entropy, causal_dist, block_row, block_col, has_outlier, seq_len, head_dim, num_heads.
- **Fitness function:** 50% accuracy (relative RMSE vs FP64 reference), 30% compression (bits saved), 20% reliability (no NaN outputs).
- **Search configuration:** 80 iterations, population of 50, 3 islands, diff-based evolution, evaluated across 6 representative workloads (sequence lengths 512-4096, head dimensions 64-128, with and without causal masking and outliers).
- **LLM backbone:** Claude Sonnet (80% weight) and Haiku (20%) generating candidate mutations via `claude_code` provider.

## Key Finding

The evolutionary search converged on a remarkably simple policy:

```python
if has_outlier and entropy < 2.0:
    return "fp16"
return "int8"
```

Out of 14 available input features, only two matter: `has_outlier` (whether the block contains values exceeding 6x the mean absolute value) and `entropy` (Shannon entropy of the softmax attention distribution). The logic: high entropy (~4.3) means attention weights are spread uniformly across keys, so outlier values are diluted and INT8 quantization error is harmless. Low entropy (< 2.0) means attention is peaked on a few keys -- if those keys happen to be outliers, quantization destroys the signal.

Evaluated across all 12 workloads (sequence lengths 512-8192):

- **58% of blocks** assigned INT8 (all non-outlier workloads compress fully).
- **42% of blocks** retain FP16 (all blocks in outlier workloads with peaked attention).
- **Evolved policy RMSE matches FP16 baseline** -- no measurable accuracy degradation.
- **Uniform INT8 baseline fails catastrophically** on outlier workloads (relative RMSE > 1000x worse).
- **Entropy distribution is bimodal:** ~14,000 blocks cluster at ~4.3 (safe for INT8), ~5,600 blocks cluster at 0.5-1.0 (dangerous if outliers present).
- **Sequence length has no effect** on the policy decision -- the same entropy + outlier test works from N=512 to N=8192.

The policy operates at effective workload granularity: because `has_outlier` correlates across all blocks within an outlier workload, the decision is all-INT8 or all-FP16 per workload. This is actually a strength -- it means the precision decision can be made once before the kernel launches, not per-block at runtime.

## Hardware Implications

The simplicity of the converged policy is a direct asset for ASIC implementation:

1. **Two comparisons, zero arithmetic.** The policy requires one boolean check (`has_outlier`) and one fixed-point threshold comparison (`entropy < 2.0`). No multiplications, no lookup tables, no floating-point logic in the decision path.
2. **Pre-kernel decision.** Since the policy operates at workload granularity, the precision mode can be set once per attention call. The kernel itself does not need per-block branching or dynamic precision switching.
3. **No feature extraction overhead.** Only two of the 14 statistics need to be computed at runtime. `has_outlier` is a byproduct of the data loading stage (max vs mean comparison). Entropy requires a softmax pass, but this is already computed as part of attention -- it can be accumulated from a previous tile pass or estimated from a small sample.
4. **Binary precision mode.** The search explored six precision levels and converged on just two (INT8 and FP16). Hardware only needs to support two datapaths, not a full precision-flexible ALU.

### Why Not a Learned Controller?

We considered replacing the threshold-based policy with a learned controller (small MLP or decision tree) that could make more nuanced per-block decisions. We chose not to, for several reasons:

1. **The search already explored complexity.** The evolutionary search had access to 14 input features, 6 precision levels, and 80 iterations of free-form code mutation. It could have evolved arbitrary decision logic -- weighted feature combinations, multi-threshold cascades, lookup tables. It converged on two comparisons because the problem is genuinely separable: the entropy gap between "safe" (~4.3) and "dangerous" (~0.5-1.0) blocks is a canyon, not a crack.

2. **Hardware cost is asymmetric.** A comparator + AND gate costs negligible area and power. Even a tiny MLP (8 weights) requires multipliers, activation logic, weight SRAM, and pipeline stages on the critical path of every attention block. The precision gain, if any, does not justify the silicon budget.

3. **Verification.** Proving a threshold comparator behaves correctly in all cases is trivial. Proving a neural network controller never makes a catastrophic precision decision is an open research problem.

4. **Programmable thresholds capture the upside.** Instead of a fixed `entropy < 2.0`, the threshold is stored in a per-layer register file (`THRESHOLD_REG[layer_idx]`). Profile a model once, find the optimal threshold per layer, program the registers. This provides per-layer adaptation with zero additional logic beyond a small register file.

```
// Hardware pseudo-logic for precision controller
precision = (has_outlier AND entropy < THRESHOLD_REG[layer_idx]) ? FP16 : INT8;
```

This gives ~95% of the adaptability of a learned controller at ~0% of the cost. The only scenario where a learned controller wins is if real model activations reveal a non-linear decision boundary in entropy space -- a hypothesis we will test during real-activation validation (see Next Steps).

## Limitations

1. **Synthetic data only.** All workloads use `torch.randn` inputs with injected outliers (0.1% of values scaled by 100x). Real LLM activations have different distributional properties -- heavy tails, structured sparsity, layer-dependent ranges.
2. **Workload-level, not block-level granularity.** The evolved policy does not make fine-grained per-block decisions within a single workload. A more nuanced policy might compress some blocks even within outlier-heavy sequences.
3. **Outlier detection depends on threshold.** The `has_outlier` flag uses a fixed rule (`abs_max > abs_mean + 10 * abs_std`). This threshold was not co-evolved and may not transfer to real activations.
4. **No latency or throughput measurement.** The fitness function measured accuracy and compression ratio, not wall-clock speedup. Actual hardware gains depend on memory bandwidth savings, which were not benchmarked.
5. **Single attention pattern.** The workloads test standard dense attention. Sparse attention, sliding window, or grouped-query attention may shift the entropy distribution.

## Next Steps

1. **Real LLM activation validation.** Run the policy on captured QKV tensors from production models (Llama, Mistral, GPT-class) to verify the entropy threshold and outlier detection transfer from synthetic to real data.
2. **KV-cache quantization.** Apply the entropy-based decision to KV-cache compression during autoregressive generation, where memory savings have direct latency impact.
3. **Per-layer entropy profiling.** Characterize which transformer layers and heads consistently produce low-entropy attention, enabling static per-layer precision assignment at compile time.
4. **Co-evolve the outlier threshold.** Include the `has_outlier` detection threshold (currently `abs_max > abs_mean + 10 * abs_std`) as an evolvable parameter alongside the entropy cutoff.
5. **Hardware prototype.** Implement the two-comparison decision circuit in RTL and measure area/power overhead relative to a fixed-precision attention datapath.
